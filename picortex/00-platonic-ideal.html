Mockup 00The platonic ideal

What an "awesome texting experience" feels like from the user's side, and the abstract architecture that must be behind it. Implementation choices (tmux vs claude -c -p, one-box vs two-box) are deferred to the concrete-option mockups — this page is what it must feel like, not how we build it.

The one test

A friend adds the bot to a group text. Over the next hour: it stays quiet during chatter. When @mentioned with a public-friendly question, it replies usefully in <15 s. When someone asks "is Jacob free Tuesday?", the bot says "let me check", DMs Jacob privately with the question + proposed response, and only answers the group once Jacob approves. No one sees private information Jacob didn't explicitly approve.

1User stories

As Jacob

I add my bot to a group chat with two friends. It answers their questions without me having to respond myself, and it knows when to stay quiet.

As Jacob's friend

I ask "is Jacob free Tuesday?" in the group. I get a real answer within a couple of minutes, not "I can't access that."

As Jacob

When the bot needs my private info to answer in a group, it DMs me with the exact response it wants to send, and I approve or tweak before anything leaves my DMs.

As Jacob

I can tell the bot "louder" or "quieter" per chat, and I can see and shape what it's allowed to know about me in each chat.

As a group member

When I react ⭐ to a useful bot message, it saves it to Jacob's notes on our behalf. ❌ retracts a bad bot answer.

As Jacob

It feels native — threads, reactions, typing indicator, no markdown walls on iMessage.

2Scene A — bot respects attention

Default in a new group chat: mentions-only. Bot stays silent through normal chatter. Speaks when @mentioned, replied-to, or slash-commanded.

3Scene B — the consent loop (the banger)

Group members ask something that requires data the group's knowledge manifest doesn't cover. Bot pauses, DMs Jacob out-of-band with the proposed response, and only replies in-group after explicit approval.

6Architecture — abstract, channel- and executor-agnostic

The minimum set of components any concrete option must instantiate. The platonic architecture only names roles, not machines or processes. See the two concrete mockups for one-box and two-box materializations of these roles.

stateless role out-of-band / privacy path user endpoint in-channel (group) flow out-of-band (private DM) flow

Why "out-of-band"? The approval question from the bot goes to Jacob's private DM with the bot — not back into the group where Alice asked. Same phone, same app, same DM thread Jacob already uses for 1:1 chats; distinct conversation from the group. The group only sees "let me check" and the eventual answer (or "I don't have access to that"). Alice cannot observe that an approval loop happened.

7Flow of a consent-loop turn

  1. Inbound. Alice's message arrives on the Channel adapter (Linq or OpenChat webhook). HMAC verified, dedup'd, logged.
  2. Attention gate. Contains @picortex and a question addressed to Jacob's calendar. Gate passes.
  3. Executor starts turn. Reads the chat's manifest — calendar not in scope.
  4. Pause → consent broker. Executor yields; consent broker takes over. Bot emits "let me check" to the group; typing indicator on.
  5. Out-of-band DM to Jacob. Consent broker formats context + proposed response; sends via the same Channel adapter to Jacob's DM.
  6. Jacob approves. Reply "y" (or tap approve). Broker validates, writes DisclosureEvent, resumes the original turn with scope expanded.
  7. Final reply. Executor finishes and emits the approved message into the group. Audit row closes with final_reply.
  8. Timeout path. If Jacob doesn't respond in 10 min, broker closes the loop: group gets "I need to check on that offline", audit row marks approval_mode: "timeout". No indefinite waits.

8What's explicitly abstract here

  • Nothing says one process or two, one box or many, tmux or not.
  • Knowledge layer is drawn as one role but in reality is { per-chat manifest store, noos graph, transcript, files } — any of which can live anywhere.
  • The Channel adapter hides Linq vs OpenChat specifics — both normalize to the same event vocabulary.
  • The Executor is the most replaceable piece: claude -c -p per turn, a tmux REPL, OpenClaw, or something else — all satisfy this diagram.

9What the concrete options change

Option 2 · No-Docker

Bot server and the Claude box live on different machines. Executor is an SSH call that runs claude --session-id <chat> -p. Knowledge layer is HTTP to noos on a third reused box. See mockup → (or the reference doc for what Piyush literally shipped)

Option 4 · Noos-style

Everything in one process, no persistent workspace. Executor rebuilds transcript per turn and calls claude --print. Knowledge layer is HTTP to noos. See mockup →
